**Дата**: 2026-04-27
**Статус**: Завершено — все вопросы закрыты

## 1. Источник начального каталога парфюмерии

**Решение**: датасет Fragrantica с Kaggle («Fragrantica Perfumes Database», ~17 000–20 000 позиций).

**Обоснование**: датасет содержит все необходимые поля — название, бренд, описание,
верхние/средние/базовые ноты, рейтинги. Загружается один раз при старте проекта через
скрипт импорта; не требует Playwright на этапе MVP. Живой скрапинг Fragrantica — только для
инкрементальных обновлений (Phase 4+).

**Рассмотренные альтернативы**:
- Парсинг Fragrantica «с нуля» — отклонено: Playwright + защита от ботов делают это
  ненадёжным для первоначального наполнения каталога.
- Ручной ввод парфюмов — отклонено: масштаб не подходит для MVP.

## 2. Алгоритм рекомендаций

**Решение**: cosine similarity на бинарных векторах нот с использованием `numpy`.

**Обоснование**:
- Каждый парфюм представлен бинарным вектором длиной N (N = общее число нот в каталоге):
  `1` если нота присутствует, `0` если нет.
- Профиль пользователя — такой же вектор из его выбранных нот.
- Схожесть: `cos(θ) = (A·B) / (‖A‖·‖B‖)`. Ранжируем по убыванию, берём топ-5.
- `numpy` уже нужен для матричных операций; подключать scikit-learn ради одной формулы
  нарушает принцип YAGNI.
- Производительность: при ~20 000 парфюмах и ~300 нотах матрица занимает ~6 МБ в памяти,
  расчёт — единицы миллисекунд.

**Рассмотренные альтернативы**:
- Коллаборативная фильтрация — отклонено: требует накопленных данных о поведении
  пользователей; неприменимо для MVP с нулевой базой.
- Эмбеддинги через LLM — отклонено: избыточно, медленно, нарушает принцип III (LLM — слой
  подачи, а не рекомендаций).

## 3. aiogram 3 FSM + Redis Storage

**Решение**: `RedisStorage` из `aiogram.fsm.storage.redis`; состояния — `StatesGroup`.

**Обоснование**:
- `RedisStorage` сохраняет состояние FSM между перезапусками бота — критично для
  производственной среды.
- `StatesGroup` — официальный способ объявления состояний в aiogram 3; обеспечивает
  статическую типизацию и автодополнение.
- Данные FSM хранятся с ключом `{bot_id}:{chat_id}:{user_id}:state`; TTL задаётся
  параметром `state_ttl` (рекомендуется 24 ч для сессий выбора нот).

**Пример структуры состояний**:
```python
class NoteSelection(StatesGroup):
    choosing_categories = State()
    choosing_notes      = State()
    viewing_results     = State()

class FavoritesMenu(StatesGroup):
    viewing_list        = State()
```

## 4. Уведомления из Celery в Telegram

**Решение**: инстанциировать `Bot(token=...)` прямо в Celery-задаче и вызывать
`asyncio.run(bot.send_message(...))`.

**Обоснование**:
- Celery-воркер и aiogram Dispatcher работают в разных процессах; делиться одним
  экземпляром `Bot` через межпроцессный канал нецелесообразно.
- Создание нового `Bot`-объекта в задаче — лёгкая операция (только конфигурация HTTP-сессии);
  токен читается из env-переменной.
- `asyncio.run()` изолирует event loop задачи от event loop бота.

**Рассмотренные альтернативы**:
- Публикация события в Redis и обработка в боте — отклонено: усложняет архитектуру; для MVP
  избыточно.
- Синхронный Telegram HTTP API (requests) — отклонено: потеря типизации aiogram, дублирование
  кода.

## 5. Стратегия скрапинга магазинов

**Решение**: `httpx` (AsyncClient) для статичных HTML-страниц; `Playwright` для
JS-рендеринга (при необходимости).

**Обоснование**:
- Randewoo, Notino, Золотое Яблоко и Летуаль на момент исследования отдают цены в
  статичном HTML; `httpx` достаточно.
- Playwright подключается только если магазин перейдёт на SPA без SSR; это решение
  откладывается до конкретной потребности (YAGNI).
- Ротация User-Agent: список из 10–15 реальных строк, выбор случайный для каждого
  запроса.
- Circuit breaker: счётчик ошибок в Redis (`scraper:{shop}:fail_count`); при ≥ 5 — установка
  флага паузы `scraper:{shop}:paused_until` на +1 ч.

**Обработка HTML-фикстур в тестах**:
- Перед написанием скрапера: сохранить реальную HTML-страницу товара в
  `fixtures/html/{shop}_product.html`.
- Тест мокает `httpx.AsyncClient.get` чтобы возвращать содержимое фикстуры.
- Это гарантирует детерминированность тестов и отсутствие живых запросов в CI.

## 6. Многосервисная архитектура Docker Compose

**Решение**: 6 сервисов в одном `docker-compose.yml`.

| Сервис | Образ | Назначение |
|--------|-------|-----------|
| `postgres` | postgres:16 | основная БД |
| `redis` | redis:7-alpine | FSM, кэш, брокер |
| `bot` | ./docker/Dockerfile.bot | aiogram 3 Dispatcher |
| `api` | ./docker/Dockerfile.api | FastAPI (uvicorn) |
| `worker` | ./docker/Dockerfile.worker | Celery-воркер |
| `worker-beat` | ./docker/Dockerfile.worker | Celery Beat (планировщик) |

Зависимости запуска: `bot` и `api` ждут `postgres` + `redis`; `worker` ждёт `redis`.
