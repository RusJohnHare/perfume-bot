# Спецификация: Парфюмерный советник MVP

**Ветка**: `001-perfume-advisor-mvp`
**Создано**: 2026-04-27
**Статус**: Черновик

## Пользовательские сценарии и тестирование *(обязательно)*

### Сценарий 1 — Выбор нот и рекомендации (Приоритет: P1)

Новый пользователь открывает бота, выбирает категории ароматов и конкретные ноты через
пошаговое меню и получает персональный список из 5 духов, наилучшим образом соответствующих
его профилю. Каждый результат включает название, бренд, краткое описание и ссылку на покупку
с актуальной ценой.

**Почему этот приоритет**: Это основная ценность продукта. Без работающих рекомендаций у
продукта нет смысла. Все остальные сценарии надстраиваются поверх этого потока.

**Независимый тест**: Пользователь без истории может открыть бота, завершить выбор нот и
увидеть минимум 5 рекомендаций с хотя бы одной ссылкой на покупку в каждой — без реализации
других сценариев.

**Сценарии приёмки**:

1. **Дано** пользователь открывает бота впервые,
   **Когда** он отправляет `/start`,
   **Тогда** бот приветствует его и показывает кнопки категорий ароматов (например: Цитрусовые, Древесные, Цветочные, Мускусные, Восточные, Свежие).

2. **Дано** пользователь выбрал хотя бы одну категорию,
   **Когда** он подтверждает выбор,
   **Тогда** бот показывает конкретные ноты внутри этих категорий для множественного выбора (например: Бергамот, Сандал, Жасмин).

3. **Дано** пользователь выбрал хотя бы одну ноту,
   **Когда** он запрашивает рекомендации,
   **Тогда** бот возвращает список минимум из 5 духов, ранжированных по схожести нот: название, бренд, краткое описание, цена в рублях и ссылка на магазин.

4. **Дано** выбранные ноты точно соответствуют менее чем 5 позициям в каталоге,
   **Когда** бот вычисляет результаты,
   **Тогда** оставшиеся позиции заполняются ближайшими частичными совпадениями с соответствующей пометкой.

---

### Сценарий 2 — Управление избранным (Приоритет: P2)

Из любой карточки рекомендации пользователь может добавить духи в личный список избранного.
Список доступен в любой момент; из него можно удалять позиции.

**Почему этот приоритет**: Избранное — обязательное условие для мониторинга цен (P3) и создаёт
повод возвращаться в бот после первой сессии.

**Независимый тест**: Пользователь может добавить духи из рекомендации в избранное, уйти,
вернуться на экран избранного и увидеть этот элемент — без активного мониторинга цен.

**Сценарии приёмки**:

1. **Дано** отображается карточка рекомендации,
   **Когда** пользователь нажимает «В избранное»,
   **Тогда** духи добавляются в список, а кнопка меняет подпись на «Убрать из избранного».

2. **Дано** пользователь открывает избранное,
   **Когда** список не пуст,
   **Тогда** бот показывает каждую позицию с: названием, брендом, последней известной ценой с датой и кнопками удаления и переключения уведомлений.

3. **Дано** список избранного пуст,
   **Когда** пользователь открывает экран избранного,
   **Тогда** бот показывает дружелюбное сообщение о пустом списке с призывом начать новую сессию подбора.

4. **Дано** духи находятся в избранном пользователя,
   **Когда** он нажимает «Убрать из избранного»,
   **Тогда** позиция удаляется из списка и мониторинг цен для неё прекращается.

---

### Сценарий 3 — Мониторинг цен и уведомления о снижении (Приоритет: P3)

Система автоматически проверяет цены всех духов из избранного в поддерживаемых магазинах
с заданной периодичностью. Когда цена снижается на 10% и более по сравнению с последней
зафиксированной, пользователь получает уведомление в Telegram с новой ценой и прямой ссылкой
на магазин. Уведомления можно включать и отключать для каждой позиции отдельно.

**Почему этот приоритет**: Мониторинг цен — «якорная» функция: она создаёт причину держать
бота активным и стимулирует конверсию в покупку. Зависит от работающего избранного (P2).

**Независимый тест**: Модератор может вручную вызвать событие обновления цены для конкретных
духов в избранном (имитация результата скрапера), и пользователь-владелец получит уведомление
в Telegram в течение 30 минут — независимо от потока рекомендаций.

**Сценарии приёмки**:

1. **Дано** у пользователя есть духи в избранном с включёнными уведомлениями,
   **Когда** система фиксирует снижение цены на 10% и более,
   **Тогда** пользователь получает уведомление: название духов, старая цена, новая цена, название магазина и прямая ссылка на покупку.

2. **Дано** у пользователя есть духи в избранном с отключёнными уведомлениями,
   **Когда** система фиксирует снижение цены,
   **Тогда** уведомление не отправляется, но обновлённая цена сохраняется и отображается в списке избранного.

3. **Дано** все источники цен для какого-либо магазина недоступны,
   **Когда** пользователь просматривает эти духи в избранном,
   **Тогда** отображается последняя зафиксированная цена с датой последней проверки; ошибка пользователю не показывается.

4. **Дано** пользователь хочет перестать получать ценовые уведомления для конкретной позиции,
   **Когда** он нажимает «Откл. уведомления» на этой позиции,
   **Тогда** уведомления отключаются только для неё; остальные позиции не затрагиваются.

### Граничные случаи

- Что происходит, если выбранным нотам не соответствует ни один парфюм в каталоге?
  → Бот сообщает, что точных совпадений нет, предлагает расширить выбор и показывает 5
  ближайших частичных совпадений как запасной вариант.
- Как система реагирует на блокировку запросов со стороны магазина?
  → Скрапер этого магазина останавливается на 1 час (circuit breaker); остальные магазины
  продолжают работать; устаревшие цены отображаются с пометкой «цена могла измениться».
- Что если пользователь добавляет одни и те же духи в избранное дважды?
  → Система автоматически убирает дубликат; в списке отображается одна запись.
- Что если пользователь удалил аккаунт Telegram или заблокировал бота?
  → Данные сохраняются, уведомления перестают доставляться; автоматическая очистка в MVP не
  предусмотрена.

## Требования *(обязательно)*

### Функциональные требования

- **ФТ-001**: Система ДОЛЖНА позволять пользователям выбирать ароматические ноты из минимум 6
  предопределённых категорий, каждая из которых содержит несколько именованных нот.
- **ФТ-002**: Система ДОЛЖНА формировать ранжированный список минимум из 5 рекомендаций на
  основе выбранных нот пользователя, используя сравнение по схожести с каталогом.
- **ФТ-003**: Каждая рекомендация ДОЛЖНА отображать: название духов, бренд, однострочное
  описание, актуальную цену в рублях (или последнюю известную цену с датой при недоступности
  данных) и ссылку на покупку.
- **ФТ-004**: Пользователь ДОЛЖЕН иметь возможность добавить любые рекомендованные духи в
  избранное прямо из карточки рекомендации.
- **ФТ-005**: Пользователь ДОЛЖЕН иметь возможность в любой момент открыть полный список
  избранного через отдельный пункт меню.
- **ФТ-006**: Пользователь ДОЛЖЕН иметь возможность удалять отдельные позиции из избранного.
- **ФТ-007**: Пользователь ДОЛЖЕН иметь возможность включать и отключать уведомления о
  снижении цены для каждой позиции избранного отдельно.
- **ФТ-008**: Система ДОЛЖНА проверять цены для всех позиций избранного с активными
  уведомлениями не реже одного раза в 6 часов.
- **ФТ-009**: Система ДОЛЖНА отправлять уведомление в Telegram, когда цена духов из избранного
  снижается на 10% и более по сравнению с последней зафиксированной ценой.
- **ФТ-010**: При недоступности актуальных ценовых данных система ДОЛЖНА отображать последнюю
  зафиксированную цену с датой и временем последней проверки, не скрывая позицию и не
  показывая ошибку.
- **ФТ-011**: Система ДОЛЖНА корректно обрабатывать выбор нот, при котором менее 5 точных
  совпадений: дополнять результаты ближайшими частичными совпадениями с явной пометкой.

### Ключевые сущности

- **Пользователь**: аккаунт Telegram, зарегистрированный в боте; идентифицируется по
  Telegram user ID; хранит отображаемое имя и дату регистрации.
- **Ароматическая нота**: именованная характеристика аромата (например, Бергамот, Сандал),
  относящаяся к категории (например, Цитрусовые, Древесные); источник — каталог парфюмерии.
- **Духи**: именованный парфюмерный продукт с брендом, описанием и набором ароматических нот;
  связан с одним или несколькими предложениями в магазинах.
- **Предложение магазина**: запись о цене и наличии конкретных духов в конкретном
  интернет-магазине; содержит цену в рублях, статус наличия, прямую ссылку и дату последней
  проверки. Записи только добавляются (append-only).
- **Избранное**: связь между пользователем и духами; хранит признак включённых уведомлений
  о снижении цены и дату добавления.

## Критерии успеха *(обязательно)*

### Измеримые результаты

- **КУ-001**: Новый пользователь завершает выбор нот и получает персональные рекомендации
  менее чем за 45 секунд с момента отправки `/start`.
- **КУ-002**: Минимум 90% сессий рекомендаций возвращают 5 результатов, каждый из которых
  содержит хотя бы одну активную ссылку на покупку.
- **КУ-003**: Проверка цен для всех отслеживаемых позиций завершается в рамках 6-часового
  интервала при штатной работе системы.
- **КУ-004**: Уведомления о снижении цены доставляются пользователям в течение 30 минут
  с момента обнаружения системой соответствующего изменения.
- **КУ-005**: Бот отвечает на все действия пользователя в течение 3 секунд для 95% запросов
  (p95 < 3 с).
- **КУ-006**: Пользователь может добавить духи в избранное и найти их на экране избранного
  в следующей сессии без потери данных.

## Допущения

- Интерфейс бота на русском языке; мультиязычность в MVP не требуется.
- Начальный каталог парфюмерии загружается из статического датасета (например, экспорт
  Fragrantica с Kaggle); живой скрапинг каталога на старте MVP не нужен.
- Поддерживаемые магазины для мониторинга цен в MVP: Randewoo, Notino, Золотое Яблоко, Летуаль.
- Пользователь идентифицируется исключительно по Telegram user ID; отдельная регистрация или
  email не требуются.
- Основной интерфейс — мобильный клиент Telegram; веб- и десктопная оптимизация не входят
  в scope MVP.
- Порог уведомления о снижении цены (10%) — фиксированная системная константа в MVP,
  настройка пользователем не предусмотрена.
- Пользователи, заблокировавшие бота или ставшие неактивными, автоматически не удаляются в MVP.
- Частота обновления каталога (ежедневное инкрементальное обновление) — операционный вопрос,
  решаемый фоновыми воркерами; пользователю не виден.
